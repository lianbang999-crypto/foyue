# 听经台长音频专项优化方案

适用范围：听经台中大量 60 分钟以上音频内容。

目标不是单纯追求“首播更快”，而是同时优化以下体验：

- 首次点播更快出声
- 拖动进度更稳定
- 中段续播更快恢复
- 长时间收听时更少卡顿
- 缓存成本可控，不把边缘和 R2 流量打爆

## 现状判断

当前系统已经具备以下基础：

- 音频域支持 Range
- Worker 已减少热点路径的 R2 双跳
- Worker 已加入安全版 edge cache
- 小 Range 请求可触发受控后台预热

这些对短音频和中等时长音频收益明显。

但对 60 分钟以上长音频，仍有三个天然限制：

- 整文件缓存成本过高
- 用户经常从中段恢复或拖动到后半段
- 首个请求即使优化了，也不能解决“整段收听过程中的稳定性”

所以长音频不能简单沿用短音频策略。

## 三档方案

## A 档：低成本优化

适合现在就做，风险低，见效快。

### 1. 听经台单独做播放意图预连接

触发时机：

- 用户进入听经台 tab
- 用户打开听经台某专辑集数页
- 用户即将点击当前集播放按钮

目的：

- 提前完成 audio.foyue.org 的 DNS / TLS / TCP 建连
- 不在首页全局抢跑

收益：

- 对首次点播有效
- 不增加太多无效流量

风险：

- 很低

### 2. 长音频降低码率

语音类听经内容通常可以比音乐更激进地压缩。

建议：

- 优先审查 60 分钟以上音频的码率
- 对热门内容先做低码率版本
- 尤其关注移动网络用户常听的长讲记

收益：

- 首播更快
- 弱网更稳
- 续播和拖动都更轻

风险：

- 需要控制音质下降幅度

### 3. 优先优化断点恢复

长音频用户更关心：

- 上次听到 43 分钟
- 这次回来能不能快点续上

建议：

- 继续强化断点保存
- 页面恢复后尽早恢复播放上下文
- 降低恢复过程中与首页/AI/文库等非播放任务的竞争

收益：

- 对长音频体验价值极高

风险：

- 中低

### 4. 不对长音频做整文件预热

保持当前策略：

- 小对象允许受控后台预热
- 长音频直接走 Range

原因：

- 全量预热长音频会显著增加 R2 和边缘压力
- 首次点播收益不一定成正比

## B 档：中成本优化

适合 A 档做完、仍然觉得长音频拖动和续播不够稳时再做。

### 1. 听经台长音频按段拆分

做法：

- 按 10 到 15 分钟切段
- 或按章节切段
- 前端仍然表现成一条内容

收益：

- 首播只拉第一段
- 拖到后半段不必背整文件
- 缓存粒度更合理

风险：

- 需要额外的播放清单和拼接逻辑

### 2. 建立长音频段清单

后端或构建阶段生成 manifest：

- 每段 URL
- 每段时长
- 段序号
- 章节信息（可选）

收益：

- 为后续播放器拼接、断点恢复、预加载提供基础

风险：

- 中等

### 3. 缓存第一段和最近段

不是缓存整文件，而是缓存：

- 第一段
- 最近续播段
- 热门讲记前几段

收益：

- 更符合真实收听行为
- 缓存投入产出比更高

风险：

- 需要额外缓存策略管理

## C 档：重构级优化

适合把听经台当成独立的大体量音频产品来做。

### 1. 听经台单独上 HLS / 流媒体分片

适用于：

- 大量 60 分钟以上内容
- 用户频繁拖动
- 网络条件差异很大

收益：

- 首播快
- 拖动稳
- 适合弱网
- 缓存粒度天然合理

成本：

- 最高
- 需要转码链路、清单生成、播放器兼容和发布链路调整

### 2. 多质量版本

例如：

- 标准版
- 移动弱网版

收益：

- 进一步提升移动端体验

成本：

- 运维复杂度显著增加

## 推荐顺序

### 第一阶段：先做

- 听经台播放意图预连接
- 长音频码率审查与压缩
- 长音频断点恢复优化
- 保持长音频不做整文件预热

### 第二阶段：效果不够再做

- 长音频分段
- manifest 清单
- 第一段 / 最近段缓存策略

### 第三阶段：体量上来再做

- HLS
- 多质量版本

## 验收标准

### A 档验收

- 长音频首播体感更快
- 断点恢复更稳定
- 不增加明显流量浪费

### B 档验收

- 拖动到中后段明显更稳
- 首播不必依赖整文件
- 缓存命中更贴近真实收听路径

### C 档验收

- 弱网和长时间收听体验明显提升
- 长音频拖动和恢复达到流媒体级体验

## 结论

对于听经台 60 分钟以上内容，最现实的路线不是直接把所有音频整文件缓存，而是：

- 先做低成本的连接、码率、恢复链路优化
- 再按需要上分段
- 最后再决定要不要进入 HLS 级别的重构

这条路径最稳，也最符合当前项目阶段。
