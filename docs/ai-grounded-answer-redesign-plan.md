# AI 问答重构方案（Grounded Answer V2）

## 1. 背景

当前 AI 问答已经具备基础检索、引用、流式返回和多模型兜底能力，但线上体验仍存在以下结构性问题：

- 用户感知上，系统有时像“原文检索”，有时像“解释问答”，有时又像“半成品引用”，模式不稳定
- 模型切换逻辑以“可用性优先”为主，而不是以“产品行为一致性优先”为主
- 引用仍主要依赖模型按文本格式输出，后端缺少强约束校验
- 提示词职责过重，一处调整容易同时影响检索、解释、引用和降级行为
- 前端虽然已经修复契约问题，但回答协议本身仍然不够清晰

这说明问题的核心不只是“模型不够强”，而是“证据协议、提示词分层和响应契约尚未被严格定义”。

## 2. 重构目标

本次重构的目标如下：

1. 让 AI 回答模式稳定可预测，避免“同类问题返回风格完全不同”
2. 让所有事实性归纳都能对应到结构化引用，而不是依赖模型自由发挥
3. 让模型提供商可以替换，但用户看到的产品行为保持一致
4. 让证据不足时的降级路径明确，不再出现“看起来回答了，其实证据不够”的情况
5. 让后台可以观察到模型、提示词版本、降级原因和引用校验结果，便于持续调优

## 3. 非目标

本次重构不追求以下方向：

- 不把产品改造成开放式闲聊 AI
- 不让 AI 在缺少原文证据时进行自由佛学推演
- 不追求一次性解决所有佛学问答边界问题
- 不优先追求更长回答，而优先追求更稳、更可核验的回答

## 4. 当前问题拆解

### 4.1 模式混杂

当前问答链路同时承担三类任务：

- 原文检索
- 依据原文做解释
- 给出修行建议

这三类任务的证据要求不同、回答格式不同、风险也不同，但当前仍在同一条主路径里混合处理。

### 4.2 提示词职责过重

当前主提示词同时负责：

- 限制模型不胡编
- 决定何时引用
- 决定何时归纳
- 决定何时只给出处
- 控制语气与格式

结果是任何小调整都可能产生连锁反应，难以单独优化某一个环节。

### 4.3 引用不可验证

当前引用更接近“模型生成的文本样式”，而不是“后端保证合法的证据对象”。

这会带来三个问题：

- 模型可能引用了不存在的资料编号
- 模型可能做出结论，但没有真正对应到某段原文
- 前端只能渲染文本，无法精确判断哪些句子是有证据支撑的

### 4.4 多模型链路缺少产品层约束

现在线上生产主链路已确认是：

- 主回答：外部 OpenAI 兼容提供商，默认模型为 GLM-4.7-Flash
- 备用回答：Groq 的 Llama 3.3 70B
- 最终兜底：Cloudflare Workers AI

问题不在于有多级兜底，而在于不同提供商的输出风格、流式行为和引用稳定性并不一致，但当前产品层要求它们产出同一种行为结果。

### 4.5 可观测性不足

当前虽然已有部分 AI 调用日志，但还缺少产品真正需要的字段：

- 当前回答属于哪种模式
- 为什么被降级
- 引用校验是否通过
- 使用了哪一版 prompt
- 检索证据强度是多少

没有这些信息，后续只能靠主观体感调优。

## 5. 目标架构

建议将问答链路重构为七段式：

1. 问题路由
2. 文档检索
3. 证据评估
4. 回答生成
5. 引用校验
6. 流式渲染
7. 最终归档与统计

### 5.1 路由阶段

路由只回答一个问题：用户这次是在要什么类型的帮助。

建议分为四类：

- quote_lookup：主要要原文出处
- grounded_explanation：希望依据原文做解释
- practice_guidance：希望获得修行上的整理式建议
- unsupported：超出资料边界或与产品定位无关

### 5.2 检索阶段

检索保持“向量 + 重排 + 元信息过滤”的基本方式，但输出必须标准化为候选证据集，而不是直接喂给模型自由发挥。

每条候选证据必须具备：

- 稳定证据编号，例如 S1、S2、S3
- 文档 ID
- 标题
- 系列名
- 原文片段
- 相关度分数
- 可选的位置偏移

### 5.3 证据评估阶段

在回答前先判断：这批证据够不够支撑回答。

建议输出三档：

- high：可以进入 grounded answer
- medium：可以给谨慎解释，但必须带强引用
- low：只允许返回 search_only

### 5.4 回答阶段

回答模型只做一件事：在已给定证据集范围内生成答案。

它不负责：

- 决定检索结果是否可信
- 自造引用编号
- 越过证据边界进行扩展推理

### 5.5 校验阶段

回答生成后，必须经过后端校验器。

如果校验失败，就不返回“看起来完整但实际上不可核验”的答案，而是自动降级为 search_only 或 partial_answer。

## 6. 产品模式设计

### 6.1 模式一：原文检索

适用场景：

- 用户要出处
- 证据不足以稳妥解释
- 引用校验失败

返回原则：

- 只给最相关资料
- 给少量引导语
- 给改问法建议
- 不做大段归纳

### 6.2 模式二：依据原文解释

适用场景：

- 检索证据强
- 问题边界清晰
- 回答可直接对应到现有资料

返回原则：

- 每个关键判断都必须对应至少一条引用
- 不允许做跨资料大跨度补全
- 句子尽量短，便于逐句核对

### 6.3 模式三：修行建议

适用场景：

- 用户明确在问“怎么落实、怎么做、如何安住”这类问题

返回原则：

- 先给原文依据，再给整理后的建议
- 显式声明“依据相关开示整理，仅供参考”
- 不把建议伪装成直引原文

## 7. 模型策略

### 7.1 推荐分工

建议采用“主回答模型 + 备用回答模型 + 辅助模型”的结构，而不是让所有模型都直接承担用户可见的最终回答。

推荐如下：

- 主回答模型：GLM-4.7-Flash
- 备用回答模型：Groq Llama 3.3 70B Versatile
- 辅助任务模型：Cloudflare Workers AI 上的 Qwen 或同级模型
- 向量模型：BGE-M3
- 语音转写：保持现有 Whisper 方案

### 7.2 为什么这样分工

- GLM 更适合作为中文主回答模型，负责风格与主质量标准
- Groq 更适合做可用性兜底，而不是定义主风格
- Cloudflare Workers AI 适合低风险、低成本、可控的小任务，例如问题改写、意图分类、简短摘要、降级提示生成

### 7.3 不建议的做法

- 不建议让三套不同提供商都直接输出用户可见的最终回答
- 不建议在没有结构化校验前，仅靠换模型来解决引用质量问题
- 不建议让低成本模型承担高风险佛学解释任务

## 8. 提示词分层设计

建议将 prompt 明确拆成四层，并分别版本化。

### 8.1 路由提示词

职责：

- 判断问题属于哪一类产品模式
- 判断是否需要先澄清
- 不直接输出最终答案

建议版本名：ROUTER_V1

### 8.2 证据提示词

职责：

- 根据检索结果评估是否足以回答
- 给出证据强度等级
- 标注为什么不够

建议版本名：EVIDENCE_V1

### 8.3 回答提示词

职责：

- 在给定证据范围内生成 grounded answer
- 每个关键句必须引用 S 编号
- 不能引用未给出的证据

建议版本名：ANSWER_V1

### 8.4 风格提示词

职责：

- 控制语言风格、长度、是否亲切、是否更解释型
- 不干预证据边界与引用规则

建议版本名：STYLE_V1

## 9. 结构化引用协议

这是本次重构的核心。

### 9.1 基本原则

- 引用编号由后端分配，不由模型命名
- 模型只能引用本次检索给出的 S1、S2、S3 等编号
- 所有事实性判断都要能回溯到 citations 列表
- 校验失败必须降级，不能带病输出

### 9.2 建议响应结构

建议最终完成态返回如下字段：

- mode
- answer
- citations
- claim_map
- follow_ups
- uncertainty
- downgrade_reason
- model_info
- prompt_version

其中 citations 每项建议至少包含：

- id
- doc_id
- title
- series_name
- quote
- score
- optional location

claim_map 每项建议包含：

- claim
- citation_ids

### 9.3 校验规则

建议后端先做确定性校验：

1. 所有 citation_ids 都必须存在于 citations 中
2. 实质性判断句必须至少挂一条 citation
3. 不允许引用未出现在本次检索结果中的资料
4. 如果回答中存在大量无引用归纳，则校验失败
5. 如果模型输出的引用格式非法，则校验失败

### 9.4 降级矩阵

- 无文档：返回 no_result
- 有文档但证据弱：返回 search_only
- 回答生成失败：返回 search_only
- 引用校验失败：返回 invalid_citation 并降级到 search_only
- 已经流出部分 token 但后续失败：返回 partial_answer，并说明仅供参考

## 10. 新的流式协议

流式接口应只承载两类内容：

- token：用于渐进渲染 answer 文本
- done：返回完整结构化 payload

要求：

- 前端不能再依赖“有时 JSON、有时 SSE”的模糊契约
- 所有成功态统一走 SSE done 收尾
- done 中必须包含完整结构化对象

这样前端就可以做到：

- 流式时先展示正文
- 完成后再补 citations、follow_ups、模式标签和免责声明

## 11. 后台可观测性

建议新增以下统计维度：

- provider
- model
- mode
- retrieval_score
- validation_result
- downgrade_reason
- prompt_version
- latency_ms

这批字段进入后台后，后续才能回答真正关键的问题：

- 哪一类问题最容易被降级
- 哪个模型最容易引用不稳
- 哪一版 prompt 改动后质量上升或下降

## 12. 分阶段实施计划

### Phase 1：回答契约重构

目标：

- 定义统一结构化响应
- 前端按结构渲染 citations 和模式标签
- done 事件完全稳定

涉及文件：

- functions/lib/ai-routes.js
- functions/lib/ai-utils.js
- src/js/ai-client.js
- src/js/ai-app.js

### Phase 2：结构化引用协议

目标：

- 检索结果标准化为 S 编号证据集
- 回答生成只允许引用 S 编号
- 引入后端校验器

涉及文件：

- functions/lib/ai-routes.js
- functions/lib/ai-prompts.js
- functions/lib/ai-utils.js

### Phase 3：提示词拆层

目标：

- 拆出 router、evidence、answer、style 四层 prompt
- 引入 prompt_version

涉及文件：

- functions/lib/ai-prompts.js
- functions/lib/ai-routes.js

### Phase 4：模型职责收窄

目标：

- 让 GLM 只做主回答
- 让 Groq 只做回答兜底
- 让 Workers AI 只做低风险辅助任务

涉及文件：

- functions/lib/ai-utils.js
- functions/lib/ai-routes.js

### Phase 5：后台监控与调优

目标：

- 后台展示 mode、provider、validation_result 和 downgrade_reason
- 支持观察 prompt 版本效果

涉及文件：

- functions/api/[[path]].js
- src/admin/view-ai.js

## 13. 验收标准

本方案完成后，至少应满足：

1. 前端不再出现“请求成功但界面卡住”的契约问题
2. grounded answer 必须全部具备可追溯 citations
3. 用户可清晰区分“原文检索”与“依据原文解释”
4. 证据不足时，系统稳定降级，不勉强回答
5. 后台可看到模型、模式、降级原因和引用校验结果

## 14. 一句话结论

这次重构的重点，不是先换更大的模型，而是先把“证据门槛、结构化引用、模式边界、响应契约”四件事做实。只要这四件事立住，提示词和模型才会真正可调；否则继续局部修补，只会反复出现“偶尔好、偶尔怪、偶尔不稳”的现象。
