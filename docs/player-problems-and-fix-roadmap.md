# 播放器问题清单与修复路线图

> 范围：仅播放器链路（不扩散到 AI / 文库）
> 目标：优先解决“弱网延迟、卡顿、页面发僵、快速切歌串写”

---

## 1. 当前主问题（按优先级）

## P0-1 弱网首播慢、切歌后长时间转圈

- 现象：弱网下切歌后缓冲时间长，用户感觉“点了没反应”
- 根因：关键路径之外还在抢占带宽（后台整曲加载、并发任务）
- 当前状态：部分已修复
- 已做动作：弱网/省流条件下跳过后台整曲加载
- 下一步：把后台任务全部改成“首播成功后再排队”

## P0-2 快速切歌时异步回调串写

- 现象：A 曲目回调晚到，覆盖 B 曲目状态
- 根因：历史上依赖分散的回调判断，缺少统一会话门禁
- 当前状态：进行中
- 已做动作：新增会话管理模块并在关键路径接入
- 下一步：将剩余异步入口统一为 session.guard 风格

## P0-3 页面偶发“发僵”或操作延迟

- 现象：播放中 UI 偶发卡顿，特别是列表和进度区
- 根因：高频事件中的重复 DOM 写入与不必要异步查询
- 当前状态：部分已修复
- 已做动作：时间更新节流、播放列表缓存标记同步化
- 下一步：继续把非关键刷新推迟到空闲时机

## P0-4 iOS 音频会话异常恢复不稳定

- 现象：来电/系统打断后恢复异常，出现“幽灵播放”
- 根因：音频元素会话可能进入异常态，简单换 src 不够
- 当前状态：已有防护
- 已做动作：切轨前强制 remove src + load 重置
- 下一步：补充更细粒度恢复埋点，统计真实恢复率

---

## 2. 现有链路（播放器核心）

1. 用户动作：点击播放/切歌
2. 会话开始：生成新的 playback session id
3. 音源准备：在线直连或离线缓存回退
4. 进入播放：ready 后执行 play
5. 稳态运行：进度更新、卡顿检测、媒体会话同步
6. 异常恢复：重试、超时兜底、错误提示

架构原则：

1. 任何异步回调都必须通过 session 校验
2. 弱网时只做关键动作，非关键任务后置
3. 播放状态以音频真实事件为准，不以意图为准

---

## 3. 已落地改造（代码层）

1. 性能热路径优化
- 进度相关 UI 更新节流，减少高频写入
- 列表渲染改为同步缓存标记，移除逐项异步缓存探测

2. 弱网保护
- 弱网/省流下跳过后台整曲加载，减少带宽竞争

3. 会话隔离
- 新增模块：[src/js/player/session.js](src/js/player/session.js)
- 接入位置：[src/js/player.js](src/js/player.js)

---

## 4. 接下来 3 步（只做播放器）

1. 完成会话门禁统一
- 将剩余 setTimeout / Promise 回调统一为 session.guard 或等价门禁

2. 提取 telemetry 子模块
- 分离播放耗时、失败原因、恢复路径统计
- 不改行为，只改可观测性

3. 提取 UI 更新子模块
- 分离进度条、状态文案、缓冲提示
- 目标是减少 player.js 主文件复杂度

---

## 5. 回归验证清单（弱网重点）

1. 2G/慢 3G 条件下首播是否明显可感知改善
2. 连续快速切歌 10 次是否出现错曲目状态
3. 播放中切前后台 5 次是否能稳定恢复
4. iOS 打断恢复后是否出现幽灵播放
5. 列表滚动和点击时页面是否仍有明显发僵

---

## 6. 判定标准（是否可进入下一阶段）

1. 首播成功率与恢复成功率稳定
2. 快速切歌不再出现串写
3. 弱网下用户可用性优先于完美体验
4. player.js 复杂度持续下降（模块化推进）
