# 音频播放简化方案

## 背景

当前播放器的复杂度主要来自多层主动优化同时存在：

- 当前曲首段预热
- 下一曲预加载
- 短音频后台整曲缓存
- Service Worker 音频缓存
- 浏览器原生流式缓冲
- Cloudflare Worker 边缘缓存

这些能力并不是全部都错，但它们叠加后会带来几个明显问题：

- 播放主链路不够直观，问题难排查
- 多个请求源可能抢占同一条播放带宽
- iPhone / iOS WebKit 上更容易出现不稳定行为
- 很难区分“传输问题”与“主动优化问题”

因此需要把系统压回一个更容易理解、更容易验证的最小模型。

## 简化目标

目标不是牺牲所有体验，而是先保证以下四点：

1. 用户点击播放后尽快出声
2. 切歌状态绝不串台
3. iPhone 上恢复、前后台切换、ghost playback 保持稳定
4. 任何播放问题都能快速定位到主链路，而不是陷在多层优化里

## 一等原则

### 音频分类

- 15 分钟内：短音频
- 15 分钟以上：长音频

这条规则先作为第一层唯一分类标准，不额外引入更多动态分支。

### 必须保留的能力

- 直接播放远程音频 URL
- 切歌会话隔离
- 用户显式暂停后，禁止自动恢复抢回播放权
- ended / next / prev / loop / shuffle 基础行为
- iPhone ghost playback 检测与恢复
- 前后台切换恢复
- Worker 正确支持 Range / 206 / Content-Range / Content-Length / CORS
- Service Worker 跳过 Range 请求，避免与流式播放冲突

说明：

- ghost playback 检测、stall 检测、前后台恢复、用户暂停意图保护，属于基础稳定性能力，不属于“可顺手删除的优化层”
- 本轮简化只删除主动预热和多层预加载，不删除恢复逻辑

### 优先删除或关闭的能力

- 当前曲额外首段预热
- 下一曲预加载
- 长音频主动并行缓存
- 多层主动优化叠加触发

## 核心判断

长音频是否边播边卡，主要取决于以下三点，而不是是否主动加热：

1. 浏览器原生流式缓冲是否正常推进
2. Range 链路是否正确稳定
3. 当前网络能否持续跟上音频码率

因此：

- 长音频不主动加热，不等于一定会卡
- 长音频主动并行缓存，也不等于一定更稳
- 很多时候，额外加热反而会引入重复请求、抢带宽和 iPhone 风险

## 三个候选方案

### 方案 A：短音频整首缓存，长音频纯流式播放

规则：

- 短音频：播放成功后，后台整首缓存
- 长音频：只使用原生流式播放，不主动预热，不并行缓存，不预加载下一首

优点：

- 模型最容易解释
- 长音频问题排查路径最短
- 对 iPhone 风险最低
- 不容易因为额外请求抢占当前播放带宽

风险：

- 某些长音频在弱网下，首播体感可能没有预热版本激进

结论：

- 这是当前最推荐的稳定优先方案

### 方案 B：短音频整首缓存，长音频仅做极轻首段 warmup

规则：

- 短音频：播放成功后，后台整首缓存
- 长音频：不缓存前 15 分钟，只允许做一个非常轻的首段 warmup，用于改善首播与初始 seek

优点：

- 比纯流式更有机会改善长音频首播体感
- 风险明显低于“主动缓存前 15 分钟”

风险：

- 仍然会在主链路外新增一层主动请求
- 仍需谨慎处理 iPhone、弱网、应用内浏览器场景

结论：

- 可作为第二阶段优化，不建议作为第一轮简化后的默认方案

### 方案 C：短音频整首缓存，长音频主动缓存前 15 分钟

规则：

- 短音频：播放成功后，后台整首缓存
- 长音频：播放开始后额外并行请求，把前 15 分钟主动拉下来

优点：

- 理论上可能改善长音频前段的连续性

风险：

- 容易与当前 audio 元素自己的流式请求形成重复下载
- 容易抢当前播放带宽
- iPhone / 弱网 / 应用内浏览器风险最高
- 代码复杂度会明显抬高

结论：

- 不建议作为当前简化方向

## 推荐方案

推荐采用方案 A：

- 短音频整首缓存
- 长音频纯流式播放

理由：

1. 这版逻辑足够简单，便于验证
2. 长音频问题不会被主动缓存行为掩盖
3. 风险主要集中在传输链路与平台兼容，边界清晰
4. 如果后续实测发现长音频首播仍偏慢，再从方案 B 增量引入极轻首段 warmup 即可

## 实施顺序

### 第一阶段：收敛主链路

- 保留直接播放
- 保留切歌状态隔离
- 保留 iPhone 恢复逻辑
- 保留短音频整首缓存
- 删除当前曲首段预热
- 删除下一曲预加载
- 禁止长音频主动并行缓存

### 第二阶段：实测验证

重点验证：

- iPhone 上长音频连续播放
- iPhone 上前后台切换恢复
- Android 上连续切歌
- 长音频冷启动起播
- 长音频 seek 后恢复
- 短音频是否能稳定整首缓存

### 第三阶段：按证据决定是否补回轻量优化

只有当实测明确证明“长音频首播体感确实不足”时，才讨论是否引入方案 B 的极轻首段 warmup。

不允许在没有证据的情况下直接恢复复杂主动缓存策略。

## 需要特别避免的误区

### 误区 1：把所有卡顿都归因于没有加热

很多卡顿其实来自：

- Range 响应不正确
- CORS 或缓存链路冲突
- 当前网络质量不足
- iPhone 音频会话异常

### 误区 2：把“多拉一点”当成稳定性的通用解法

在移动端，额外请求经常不是帮助，而是竞争。

### 误区 3：在线播放与离线缓存混为一谈

在线播放目标是尽快稳定出声。
离线缓存目标是后续复用。

这两者应尽量解耦，而不是互相绑死在同一个主流程里。

## 真实数据验证

### 音频文件大小分布

| 分类 | 时长 | 实际文件大小 | 15min 分界线 |
|---|---|---|---|
| 短音频 | <15 min | 大部分 <10MB，最大约 30MB | ✅ 安全区 |
| 长音频 | 60-120 min | 80-120MB | ⚠️ 风险区 |

15 分钟分界线精确地把小文件（<30MB）和大文件（80-120MB）分开，中间几乎没有灰色地带。

### 码率与网速关系

| 格式 | 典型码率 | 所需最低网速 | 4G 弱网（1-5 Mbps）是否足够 |
|---|---|---|---|
| MP3 128kbps | ~16 KB/s | ~128 Kbps | ✅ 通常足够 |
| MP3 192kbps | ~24 KB/s | ~192 Kbps | ✅ 通常足够 |

音频流式播放对平均带宽需求并不高，浏览器原生 buffer-ahead 在大多数正常网络下通常足以支撑连续播放。

但这不等于稳定性可以只靠平均带宽保证，仍需保留以下兜底：

- Range 链路正确
- stall 检测与恢复
- iPhone 音频会话恢复
- 前后台切换后的恢复逻辑

### 弱网下预加载反而有害的原因

```
弱网带宽 = 500 Kbps（总共可用）

无预加载时：
  audio 元素流式请求 = 占 128 Kbps → 够用，流畅播放

有预加载时（当前系统）：
  audio 元素流式请求 = 需要 128 Kbps
  后台整曲缓存请求 = 抢走 300 Kbps
  → audio 只剩 ~200 Kbps，可能出现卡顿
```

### 当前系统长音频 warmup 的实际下载量

一首 120MB / 120 分钟的音频：
- warmup ratio = min(1, 15/120) = 0.125
- windowBytes = 120MB × 0.125 = 15MB
- 实际 warmup = max(64KB, min(15MB, 24MB)) = **15MB**

播放开始前就预拉 15MB，加上 audio 元素自己的流式请求 = 双倍带宽消耗。

## 切歌冲突分析

### 所有切换场景评估

| 切换场景 | 当前系统需取消的并行请求 | 方案 A 需取消的并行请求 |
|---|---|---|
| 长→短 | warmup + bgLoad + preload + 旧流 (4个) | 仅旧流 (1个) |
| 短→长 | bgLoad + 新建 warmup + bgLoad (3个) | 仅 bgLoad + 新建流 (2个) |
| 长→长 | warmup + bgLoad + preload + 旧流 (4个) | 仅旧流 (1个) |
| 短→短 | bgLoad + 旧流 (2个) | bgLoad + 旧流 (2个) |

**结论：方案 A 在所有切换场景下并行请求更少，冲突风险更低。**

### 防竞态机制（保留不动）

- `_playCurrentId` 单调递增 callId — 所有异步回调检查是否过期
- `isSwitching` + 超时保护 — 防止切歌卡住
- `cleanupPlaybackPreparations()` — AbortController.abort() 取消所有并行请求
- 80ms debounce — 快速连续点击合并
- `dom.audio.removeAttribute('src')` — iOS 完全重置音频会话

## 代码审计发现的问题

### P0 — 必须在方案 A 中同步修复

**问题 1：SW 音频缓存绕过 playback-policy**

```js
// sw.js: 所有通过 SW 的 200 音频响应会被自动缓存
if (response.status === 200) {
  cache.put(event.request.url, response.clone());
}
```

即使 playback-policy 说"长音频不缓存"，audio 元素发不带 Range 的请求时，SW 仍会把 80-120MB 的完整响应存入缓存。**必须在 SW 层面增加过滤，只缓存已被主线程明确允许的 URL。**

**修复方案**：SW 不再自动缓存任意音频响应。短音频是否允许缓存，由主线程在 playback-policy 判定后显式触发，SW 如需参与，也只能接受主线程明确下发的允许缓存 URL 白名单，不能依赖缓存结果反推缓存许可。

### P1 — 建议一起修复

**问题 2：`togglePlay()` 恢复播放时重复触发缓存调度**

用户暂停后恢复播放，`scheduleCurrentTrackLoading()` 会被再次调用。虽然有 `bgFetchUrl` 去重，但 warmup（当前系统）没有去重机制。方案 A 删除 warmup 后此问题自然解决，但 bgLoad 的重复检查仍应保留。

**问题 3：`onVisibilityResume()` 的 `currentTime = ct` seek trick**

在 iOS Safari 上，设置 `currentTime = 当前值` 可能中断当前 buffer → 重新发 Range 请求。建议改为仅启动 stall watch，让浏览器自行恢复。

### P2 — 后续可优化

**问题 4：LRU 淘汰全量扫描 Cache API**

每次 `cacheAudio()` 后遍历所有缓存条目计算总大小。建议改为 store 维护 `totalCachedBytes` 增量计数。

**问题 5：warmedAudioUrls Set 无限增长**

`audio-url.js` 中 `warmedAudioUrls` 只 add 不 delete。方案 A 删除 warmup 后此问题自然消失。

### 行为边界说明

- 在线播放主链路目标是尽快稳定出声
- 短音频离线缓存目标是后续复用和离线回退
- 长音频默认不进入主动缓存主流程
- 恢复播放与离线回退能力保留，不属于本轮简化删除范围

## 最终一句话版本

把播放器收缩为：

- 短音频整首缓存
- 长音频纯流式播放
- 保留 iPhone 恢复和正确 Range 链路
- 删除会抢带宽、引入竞态的主动预热和多层预加载

## Cloudflare 与旧系统联动检查清单

新系统切换时，不能只改前端播放器逻辑，还必须确认旧链路中是否仍有代码会继续影响实际播放行为。

### 一、音频 Worker

目标：保持职责纯粹，只负责稳定的流式音频分发。

必须检查：

- 是否仍正确支持 Range 请求
- 是否仍正确返回 206 / Content-Range / Content-Length
- 是否仍保留正确的 CORS / Accept-Ranges 头
- 是否存在与旧前端 warmup / preload 强耦合的行为假设
- 是否还有不必要的边缘缓存逻辑会放大长音频并行请求

结论：

- 音频 Worker 是播放主链路的一部分，不能忽略
- 但在方案 A 下，它的职责应更简单，而不是更复杂

### 二、Pages Functions

目标：继续稳定输出音频播放所需的数据，不参与复杂播放策略。

必须检查：

- API 是否仍返回正确的音频 URL
- duration / bytes / mime / etag 字段是否继续保持兼容
- 是否有依赖旧前端预热逻辑的特殊字段或分支

结论：

- 如果只是落地方案 A，Functions 通常不需要大改
- 应优先保持数据接口稳定，避免把前端简化扩散成后端重构

### 三、Service Worker

目标：不绕过新策略，不继续偷偷执行旧缓存行为。

必须检查：

- 是否还会自动缓存任意音频响应
- 是否可能让长音频在新策略下仍被缓存
- 是否仍正确跳过 Range 请求
- 是否需要通过版本升级强制替换旧 SW

结论：

- SW 是新旧系统切换时最容易留下旧行为残留的一层
- 如果前端已简化而 SW 未同步修改，旧系统行为仍可能继续生效

### 四、旧缓存与迁移

目标：避免新旧策略共存导致行为错乱。

必须检查：

- 旧版本缓存中是否已经存在长音频整曲缓存
- 新策略上线后，是否需要清理旧缓存或 bump 缓存版本
- 已缓存短音频是否仍能正常离线回退
- 恢复播放是否会误命中旧策略留下的缓存结果

结论：

- 新旧缓存规则切换时，必须明确是否需要版本升级或迁移清理
- 否则“代码已改、行为未改”的问题很容易出现

本轮已落实：

- 音频离线缓存命名空间已从 `audio-v2` 升级到 `audio-v3`
- 目的不是改变短音频缓存能力，而是强制切断旧版本可能残留的长音频整曲缓存
- 这样可以避免旧缓存继续命中新策略，导致“看起来已改为纯流式，实际仍在吃旧缓存”的假验证结果

### 五、实施优先级

#### 必查且可能要改

- 前端播放器逻辑
- Service Worker
- 音频 Worker 的职责边界
- 缓存版本与旧缓存迁移策略

#### 尽量少动，优先保持兼容

- Pages Functions
- 音频 URL 构造
- API 返回的音频元数据字段

### 六、切换前最终核对

上线前至少核对以下问题：

1. 长音频是否已经完全停止主动预热和预加载
2. SW 是否仍可能自动缓存长音频
3. 音频 Worker 是否仍稳定支持原生流式播放
4. 短音频整首缓存是否仍能工作
5. 离线回退是否只作用于短音频缓存
6. 旧缓存是否会影响新策略验证结果
