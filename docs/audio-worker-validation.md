# 音频 Worker 上线后验证清单

适用范围：audio.foyue.org 的 Worker 优化上线后，验证以下两类改动是否生效：

- 去掉热点路径的 R2 head + get 双跳
- 为完整 200 响应增加安全版 edge cache

## 目标

确认以下事项：

- 音频仍可正常播放
- Range 语义没有回退
- 热门音频二次访问明显变快
- 404 / 416 / HEAD 等边界行为没有被打坏

## 验证样本

优先选 2 到 3 个真实热门音频：

- 首页佛号一条
- 有声书长音频一条
- 听经台讲记一条

每条样本都记录：

- URL
- 文件大小
- 所属桶
- 首次请求时间
- 第二次请求时间

## 验证项

### 1. 基础可用性

- 打开生产站首页，点击首页佛号，确认可以正常出声
- 进入有声书，打开一个专辑并播放一集，确认可以正常出声
- 进入听经台，打开一个专辑并播放一集，确认可以正常出声
- 切歌、暂停、继续播放，确认没有新增异常

通过标准：

- 无新增播放失败
- 无新增卡住转圈
- 无新增错误跳回首页或空白页

### 2. HEAD 行为

- 对真实音频发 HEAD 请求
- 确认状态码是 200
- 确认有 Content-Length
- 确认有 Content-Type: audio/mpeg
- 确认没有错误返回 body

通过标准：

- HEAD 正常返回元信息
- 不触发异常下载或多余正文传输

### 3. Range 行为

- 对真实音频发 Range: bytes=0-1 请求
- 确认状态码是 206
- 确认 Content-Length 与返回体长度一致
- 确认播放器在生产环境仍能从头开始播放

通过标准：

- 206 正常
- 播放器首播无回退

### 4. 404 / 416 边界行为

- 请求一个不存在的音频 URL，确认返回 404
- 请求一个非法 Range，确认返回 416

通过标准：

- 404 不误返回 200
- 416 不误返回正文

### 5. 热点音频二次访问速度

对同一音频，在同一区域、同一网络环境下连续测两次：

- 第一次完整 GET
- 第二次完整 GET

如果条件允许，再测：

- 完整 GET 之后的 Range 请求

重点记录：

- TTFB
- 总耗时
- 是否出现明显下降

通过标准：

- 第二次完整 GET 明显快于第一次
- 完整 GET 之后的 Range 请求不慢于改动前基线

### 6. 缓存命中观察

若生产日志或响应头能看到缓存相关信号，则额外记录：

- 是否出现缓存命中标记
- Age 是否增长
- 同一机房重复请求是否明显加速

如果看不到明确缓存命中头，也至少记录：

- 同区域第二次访问耗时变化

### 7. CORS 行为

- 从生产域 https://foyue.org 发起请求，确认音频可正常加载
- 确认本地 localhost 仍可能因为 CORS 限制而无法直接 fetch 音频，这是预期，不视为本次回归

通过标准：

- 生产站音频正常
- 本地限制不误判为线上故障

## 建议记录格式

每条样本至少记录一行：

- 音频名
- 首次 GET 耗时
- 第二次 GET 耗时
- 首次 Range 耗时
- 第二次 Range 耗时
- 是否播放正常
- 备注

## 验收结论模板

### 通过

- 播放链路正常
- HEAD / Range / 404 / 416 均正常
- 热点音频重复访问明显提速

### 有保留通过

- 播放链路正常
- 功能边界正常
- 加速效果有限，需要继续查边缘命中或首个 Range 请求策略

### 不通过

- 播放异常
- Range 回退
- 404 / 416 错误
- 重复访问没有任何改善且日志显示仍频繁回 R2
